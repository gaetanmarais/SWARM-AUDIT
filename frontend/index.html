ARCIS-SWARM — How it works

Reference guide for the discovery process, audit script, role detection, and AI analysis modules.

1. Discovery

Discovery is manual: you add servers and credential profiles in the Credentials tab. Each server is associated with a credential profile (or the default profile). The backend stores the inventory in inventory.json.

When an audit is launched, the backend connects to each server via SSH (using paramiko), uploads the audit script to /tmp/arcis_audit_*.sh, executes it, and streams back the JSON payload. The script is self-contained — it requires only bash, ss, ip, and standard Linux utilities.

Additionally, on Gateway/SCS nodes, the backend also queries the Elasticsearch API directly (_cat/health, _cat/nodes, _cat/indices, _nodes/stats, _cat/allocation) to enrich the ES audit data without relying solely on SSH.

2. Audit Script (scripts/audit.sh)

The script runs on each server and collects the following data blocks:

System specs

  • CPU count & model (/proc/cpuinfo)
  • RAM total & available (/proc/meminfo)
  • Disk usage per partition (df -BG)
  • Hostname, OS release, kernel, uptime

Network

  • All interfaces & IPs (ip addr show)
  • Active TCP/UDP connections (ss -tupn) — remote IPs, ports, process names
  • Listening ports per service (ss -tlnp / ss -ulnp)

Config files (read & content)

  • /etc/haproxy/haproxy.cfg
  • /etc/keepalived/keepalived.conf
  • /etc/caringo/cloudgateway/gateway.cfg
  • /etc/caringo/csn/conf.d/csn.cfg
  • /var/opt/caringo/netboot/content/cluster.cfg
  • /etc/caringo/scspproxy/scspproxy.cfg
  • /opt/caringo/storageui/conf/storageui.cfg
  • /opt/caringo/contentportal/conf/contentportal.cfg
  • /opt/caringo/listingcache/conf/listingcache.cfg
  • /opt/caringo/swarmfs/conf/swarmfs.cfg
  • /etc/caringo/node.cfg / cluster.cfg
  • /opt/caringo/castor/conf/castor.cfg
  • /var/opt/caringo/node/node.cfg
  • /etc/foundationdb/fdb.cluster
  • /etc/elasticsearch/elasticsearch.yml
  • /etc/rabbitmq/rabbitmq.conf
  • /etc/prometheus/prometheus.yml, alertmanager, rules
  • /etc/grafana/grafana.ini
  • /etc/chrony.conf / /etc/ntp.conf
  • /etc/rsyslog.conf + rsyslog.d/*.conf
  • /etc/syslog-ng/syslog-ng.conf
  • /etc/dhcp/dhcpd.conf

SCS — scsctl virtual sources

  • scsctl://storage/config — cluster-wide storage params (from etcd)
  • scsctl://platform/config — SCS platform params
  • scsctl://storage/cluster — live cluster topology (nodes, domains, streams)

Castor storage nodes

  • SNMP healthreport per node via swarmctl -Q healthreport (from GW/SCS node)
  • Feed list via swarmctl -Q feeds (from GW/SCS node)
  • Node list via swarmctl -Q nodes

Logs (last 24h, 100 KB cap)

  • HAPROXY: journald haproxy → /var/log/haproxy.log
  • CONTENT_GATEWAY: journald caringo-gateway → /var/log/datacore/gateway*.log
  • LISTING_CACHE_SERVER: same as gateway
  • ELASTICSEARCH: journald elasticsearch → /var/log/elasticsearch/*.log
  • CASTOR: journald castor → /var/log/caringo/castor.log
  • SCS: journald scs → /var/log/datacore/scs*.log
  • LISTING_CACHE (Redis): journald redis-server → /var/log/redis/*.log

Fallback order: journald → date-filtered file lines → tail 5000 lines. High-volume logs are deduplicated by message frequency.

3. Role Detection

Roles are detected deterministically — no heuristics. Each role requires at least one positive match. Multiple roles can co-exist on the same node.

HAPROXY Process haproxy running OR service haproxy active
CONTENT_GATEWAY Service cloudgateway / caringo-gateway active OR (package caringo-gateway installed AND gateway.cfg present) — skipped if node is an LCS
S3 Service cloudgateway active (the S3 API endpoint is the gateway daemon itself)
CONTENT_UI Package caringo-gateway-webui OR service contentportal active OR contentportal.cfg present OR gateway.cfg section [scsp] with enabled = true
STORAGE_UI Service storageui active OR storageui.cfg present OR (package caringo-storage-webui AND gateway.cfg present) OR gateway.cfg section [cluster_admin] with enabled = true
LISTING_CACHE_SERVER RabbitMQ indicators: process rabbitmq / beam.smp, service rabbitmq-server, package rabbitmq, port 5672 (AMQP) or 15672 (management) — OR service listingcache / package caringo-listingcache
LISTING_CACHE Process redis-server, service redis active, port 6379 listening, OR package redis-server
SCS Binary scsctl present OR service csn / scs / caringo-csn active OR package caringo-csn / caringo-scs OR csn.cfg / cluster.cfg present OR port 8095 (SCS API) OR ports 2379+2380 (etcd cluster)
ELASTICSEARCH Package caringo-elasticsearch-search OR process java with elasticsearch in command line OR port 9200 listening OR elasticsearch.yml present
STORAGE_NODE (Castor) Process castor running OR service castor active OR package castor / caringo-castor OR castor.cfg / node.cfg / /var/opt/caringo/node present OR port 80/443 on a non-GW node
Note: Castor nodes are typically not SSH-reachable. Their config is collected indirectly from SCS/GW nodes via scsctl and swarmctl.
FONDATION_DB Process fdbserver running OR service foundationdb active OR /etc/foundationdb/fdb.cluster present
SWARMFS Process swarmfs running OR service swarmfs active OR swarmfs.cfg present

4. AI Analysis Modules

After each audit, Claude (Haiku model) analyzes each role group independently, then runs a cross-role coherence synthesis. Each module receives config files, logs, and optional extended data specific to its role.

HAPROXY

Config: haproxy.cfg, keepalived.conf — backend server lists, balance algorithms, health check timeouts, VRRP configuration

Logs: journald haproxy → /var/log/haproxy.log — backend errors, connection resets, health check failures

CONTENT_GATEWAY / S3 / CONTENT_UI / STORAGE_UI

Config: gateway.cfg — SCSP cluster IPs, S3 settings, listing cache endpoints, webUI enabled sections ([scsp], [cluster_admin])

Logs: journald caringo-gateway → /var/log/datacore/gateway*.log

LISTING_CACHE_SERVER

Config: rabbitmq.conf, listingcache.cfg

Logs: /var/log/datacore/lcs*.log, fallback to gateway log

SCS / Castor

Config: csn.cfg, cluster.cfg, scsctl://storage/config, scsctl://platform/config, scsctl://storage/cluster (live topology from etcd)

Logs: journald castor + journald scs → /var/log/caringo/castor.log, /var/log/datacore/scs*.log

ELASTICSEARCH

Config: elasticsearch.yml — cluster name, node roles, memory lock, security settings

ES API (via HTTP): _cat/health, _cat/nodes?v, _cat/indices?v&h=…, _nodes/stats, _cat/allocation?v

Logs: journald elasticsearch → /var/log/elasticsearch/*.log

FEED (synthetic module)

Data source: SNMP healthreport Feed Table aggregated across all storage nodes (collected from SCS/GW via swarmctl -Q healthreport)

Extended: swarmctl -Q feeds — feed definitions (name, state, type, streams/min, queue depth, failure causes)

Note: FEEDs are not servers — this module analyzes data replication flow health across the cluster

Coherence Synthesis

After all per-role modules complete, a final pass receives a digest of all findings and looks for cross-role inconsistencies, configuration mismatches between layers, and global risk patterns that individual modules would miss.
